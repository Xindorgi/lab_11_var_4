## Коммит 5 — H4: Использовать docker buildx для кросс-платформенной сборки (arm64/amd64)

### Промпт 1

**Инструмент:** Claude  
**Промпт:**

> Нужно настроить кросс-платформенную сборку через docker buildx. В Go-приложении уже есть ARG TARGETOS и TARGETARCH. Подскажи команды для сборки под linux/amd64 и linux/arm64.

**Результат:** Получены команды для настройки buildx: создание builder, сборка с `--platform linux/amd64,linux/arm64`.

### Промпт 2

**Инструмент:** Claude  
**Промпт:**

> Пытаюсь собрать multi-arch образ локально: `docker buildx build --platform linux/amd64,linux/arm64 -t go-app:multi -f Dockerfile . --load`. Ошибка: `--load` не работает с несколькими платформами. Как быть?

**Результат:** Объяснение: для локального использования нужно собирать под каждую платформу отдельно с `--load`, либо использовать `--push` в registry. Выбрал вариант с отдельной сборкой для демонстрации.

### Промпт 3

**Инструмент:** Claude  
**Промпт:**

**Результат:** Показана команда `docker inspect` с фильтром по Architecture. Создан файл `results/buildx-success.txt` с логами сборки.

## Коммит 6 — H6: Реализовать graceful shutdown и проброс сигналов в контейнеры

### Промпт 1

**Инструмент:** Claude  
**Промпт:**

> Нужно протестировать graceful shutdown для обоих приложений. Как правильно отправить SIGTERM контейнерам и увидеть, что они завершаются корректно?

**Результат:** Предложено использовать `docker compose stop` (отправляет SIGTERM) и смотреть логи. Объяснено, что `docker kill` или `docker stop` с таймаутом тоже подходят.

### Промпт 2

**Инструмент:** Claude  
**Промпт:**

> Добавь в compose.yaml `stop_grace_period: 10s` для обоих сервисов. Это же влияет на graceful shutdown?

**Результат:** Подтверждение, что `stop_grace_period` задаёт время ожидания перед принудительным SIGKILL. Добавлено в compose.yaml.

### Промпт 3

**Инструмент:** Claude  
**Промпт:**

> Запустил `docker compose stop`, в логах Python вижу `Shutting down`, `Application shutdown complete`. А в логах Go только `exited with code 0`, без явных сообщений о shutdown. Это нормально?

**Результат:** Сказано, что для демонстрации лучше добавить логирование. Предложен код с `log.Printf` при получении сигнала и завершении.
Добавил логи в Go. Теперь вывод такой:
```
python-app  | INFO:     Shutting down
python-app  | INFO:     Waiting for application shutdown.
python-app  | INFO:     Application shutdown complete.
go-app      | 2026/04/09 16:26:49 Received signal: terminated, initiating graceful shutdown...
go-app      | 2026/04/09 16:26:49 Server gracefully stopped
python-app  | INFO:     Finished server process [1]
```

### Итого

- **Количество промптов:** 3
- **Что пришлось исправлять вручную:** добавление логов в Go для наглядности
- **Время работы:** ~25 минут
